## 設計原則の違反：danger_level の LLM 生成（Phase 1 から継続）

**発覚日:** 2026-05-20
**該当コミット:** 7b0ef55（2026-05-17、phase 1 done）

### 何が問題だったか

「判断は人間がやる」という設計原則があったにもかかわらず、
Phase 1 の最初から LLM に danger_level を生成させていた。

danger_level は HITL 確認の要否を決めるフィールドである。
これを機械（LLM または Safety Checker）が決定している時点で、
人間ではなく機械が判断していることになり、設計原則に反する。

Safety Checker が LLM の出力を上書きするとしても、
機械が自動的に「このステップは確認不要」と判断して実行している。

### 副作用

- SYSTEM_PROMPT の全 JSON 例に danger_level が含まれ、プロンプトが肥大化した
- LLM が danger_level の生成に注意を使い、本来重要なフィールドの精度が下がった
- LLM が danger_level を欠落させることでバリデーションエラーが頻発した

### 次の実装でやること

1. LLM に danger_level を生成させない
   - JSON スキーマから danger_level を削除する
   - SYSTEM_PROMPT の全例から danger_level を削除する

2. HITL 確認は全ステップで必ず行う
   - 「このステップは安全だから確認不要」という自動判断をなくす
   - 確認をスキップするかどうかは人間が --yes フラグで明示的に指定する

3. Safety Checker の役割を「実行可否の判定」に限定する
   - danger_level の判定ではなく、blocklist・パス制限・禁止操作の検出のみを行う
   - 「危険かどうか」の判断は人間がする



# OpenPaw Desktop v1 — 失敗の記録

> このドキュメントは v1 の設計・実装が失敗に終わった理由を記録する。
> 次の設計者（または次のセッションのClaude）が同じ過ちを繰り返さないための文書である。

---

## 何を作ろうとしたか

qwen2.5:7b（Ollama、ローカル完結）を使い、自然言語でLinuxデスクトップを操作するエージェント。
ユーザーが「古いPDFをDocumentsに移動して」と入力すると、LLMがJSONプランを生成し、
Safety Check → HITL確認 → 実行 という流れで動く。

ツールは shell / filesystem / dbus / gui / atspi の5種類。

---

## なぜ失敗したか

### 1. qwen2.5:7b に複雑な JSON 生成を任せすぎた

LLM に以下を同時に要求していた：

- 複数ツールから正しいものを選ぶ
- JSON スキーマを正確に守る（step_id, description, danger_level, on_error 等の必須フィールド）
- ツール固有のフィールドを正しく使う（filesystem の action/src/dst、dbus の service/object/method 等）
- 時間条件・複合条件を正しく解釈する

qwen2.5:7b はこれを安定して行う能力がない。
必須フィールドの欠落、存在しないメソッド名の生成、誤ったツール選択が頻発した。
非決定的であるため、同じ入力でも成功したり失敗したりする。

### 2. より大きなモデルへの移行は不可能

qwen2.5:14b は品質が上がるが応答速度が実用に耐えない。
ローカル完結という制約のもとでは、高品質なモデルと実用的な速度を両立できなかった。

### 3. SYSTEM_PROMPT の肥大化がモデルの性能をさらに悪化させた

ツールが増えるにつれ SYSTEM_PROMPT が長くなり、7b モデルの注意力が分散した。
dbus の二段階プランニング（introspect → メソッド一覧 → プラン生成）、
atspi の追加、DE 固有設定の動的ロードと、複雑さが際限なく膨らんだ。

### 4. アーキテクチャの複雑さがデバッグを困難にした

- dbus 二段階プランニング: introspect が SSH 環境で動かない
- atspi: 実際のユースケースに存在しない
- DE 固有設定: XDG_CURRENT_DESKTOP が SSH で未設定

環境依存の問題が重なり、どこで何が失敗しているか特定が困難だった。

### 5. 実際のユースケースと乖離したテストを行った

「KRunnerを開いて」のような実際には使わないタスクでテストを重ねた。
動くべきもの（ファイル操作）が動く状態で、動かないもの（複合条件、GUI制御）を
無理に動かそうとして時間を浪費した。

### 6. 機能追加が設計方針の逸脱を招いた

Phase 4 で atspi を「クロスDE対応の本道」として追加したが、
これは当初の設計方針（シングルDE、GUI依存最小化）と矛盾していた。
追加・修正・削除・巻き戻しを繰り返し、最終的に多くの時間を無駄にした。

---

## 実際に動いたもの

v1 で確実に動作したのは以下のみ：

| タスク | ツール |
|--------|--------|
| ファイル一覧 | shell |
| ファイル移動・削除・コピー | filesystem |
| 条件付きファイル操作（`$prev`） | shell + filesystem |
| スクリーンショット | gui |

これらは LLM への要求が単純で、生成される JSON の構造が限定的だったため安定した。

---

## 教訓

1. **qwen2.5:7b に JSON スキーマの厳密な遵守を期待してはいけない**
   - フィールド数が多いほど欠落が増える
   - ツールの種類が多いほど誤選択が増える

2. **ローカル7bモデルに任せるなら、生成物を極限まで単純にする**
   - 複雑な JSON より、単純なテキスト（bash コマンド1行など）の方が信頼できる

3. **動くものを先に確定させ、動かないものを追加しない**
   - ファイル操作が動く状態で dbus・atspi を追加したことで、動くものまで壊れた

4. **テストは実際のユースケースで行う**
   - 開発者が使いやすいテストケース（KRunner等）ではなく、
     ユーザーが実際に使うタスクでテストする

5. **速度と品質のトレードオフを最初に確定させる**
   - 14b は遅すぎる、7b は不安定、という制約を最初に文書化し、
     設計に反映すべきだった

---

## 次の設計への示唆

v1 の失敗を踏まえると、次の設計では以下を検討すべきである：

- LLM に生成させるものを極限まで単純にする（bash コマンド1行など）
- ツールの種類を絞る（または撤廃する）
- 生成物の検証をコードレベルで厳密に行う
- 実際のユースケースでの動作確認を設計の最初の段階で行う
